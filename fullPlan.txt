Carmel–Kinneret Trail — Community Ecosystem (Mobile + Functions + Admin Web)
Context
עמותת "כרמל כנרת" runs a physical trail connecting people (mother–daughter focus). The hackathon goal is one Firebase-backed ecosystem — a single data world reflected across a mobile app and an admin web dashboard, gated only by Role (User vs Admin):

User (hikers/runners): background geofence missions (NFRs), photo → community feed, virtual torch relay (pass the torch to a stranger ahead), hidden admin GPS/audio alerts, take-home actions matched to stations visited, always-on SOS → 101.
Admin (God Mode): full-screen live map with moving pins for phones AND IoT shirt sensors (ingested via an HTTP API), place NFRs live, fire GPS alerts, define take-home messages, leaderboard (runner speeds + community km).
Backend (אופק/אביב): Cloud Functions raise/normalize the data and enforce anti-cheat so nobody fakes km or DoS's the server.
This plan builds all three surfaces with up-to-date 2026 libs, fully in TSX.

Maps (BOTH libraries — DECIDED):
react-native-maps — the primary, rich interactive map (user trail map + admin God-Mode): custom animated logo/value markers, per-type marker views, the synced AmazingCardSlider-style carousel, viewport culling, animateToRegion, alert-radius circles — i.e. everything in the "Map UX" section that needs full custom-view control. Uses the existing Google Maps API key on both iOS & Android for visual parity. Source: react-native-maps (Expo).
expo-maps (~56.0.7, ALPHA) — Expo's first-party map for preview/static maps (station detail-sheet "show on map", profile, share cards). Read the bundled SDK doc (expomapsdk.txt); key facts that shape usage:
Separate components per platform: AppleMaps.View (iOS) / GoogleMaps.View (Android) — switch on Platform.OS. Apple Maps on iOS (no key), Google Maps on Android (needs the Android Maps SDK key + Google Cloud setup), iOS 17+.
Markers are a declarative markers={[...]} array — NOT custom child views. A marker takes coordinates, title, id, and icon (a SharedRef<'image'>); Apple additionally supports systemImage/monogram/tintColor. So per-type marker icons are done via prebuilt icon images (our assets/markers/), and value/torch/hiker styling is encoded in those icons — no animated React views on the marker here.
Interaction is callback + imperative: onMarkerClick, onMapClick, onCameraMove; ref.selectMarker(id,{moveCamera,zoom}) and ref.setCameraPosition(...). Supports circles (alert radii), polylines (trail route), polygons, uiSettings, properties (isMyLocationEnabled), color scheme.
Built-in location-permission config plugin (requestLocationPermission, locationPermission) — set the Hebrew NSLocationWhenInUseUsageDescription here too.
Alpha caveat: expect breaking changes; we pin the version and keep expo-maps behind the abstraction so a breaking release can't ripple through the app.
src/map/ abstraction wraps both behind one interface (Marker, Region, onPress, selectMarker, setCamera) so each screen picks a backend without leaking specifics: rich screens → react-native-maps (animated custom markers + carousel), preview/static → expo-maps (declarative). Set iOS deployment target 17 (expo-build-properties) and the Google Maps key + Google Cloud "Maps SDK for Android" for both.
Location & geofencing: expo-location (foreground + background updates, geofencing) + expo-task-manager (background task + startGeofencingAsync for station/alert regions).
Deep linking: expo-linking (custom scheme + Universal/App Links) — every expo-router path is deep-linkable (see Deep linking section); links from pushes, web, and external sources.
Internet / connectivity: expo-network + @react-native-community/netinfo (detect online/offline, gate server calls, offline banner) — INTERNET/ACCESS_NETWORK_STATE perms.

functions/src/ modules (all TS):

index.ts — re-exports.
admin/setAdmin.ts — onCall setAdmin (caller must already have role:admin; enforceAppCheck) → setCustomUserClaims(uid,{role:'admin'}). Plus a one-off scripts/bootstrap-admin.ts (Admin SDK) to grant the first admin.
iot/ingest.ts — onRequest HTTP API POST /ingest: IoT shirt sensors push {deviceId, lat, lng, speed, ts, secret}. Validates a shared secret/HMAC, rate-limits, writes to RTDB live/{deviceId} with source:"sensor". (The "dumb sensor = same pin as phone" requirement.)
torch/relay.ts — onCall takeTorch (proximity + transaction lock) and dropTorch (validate segment km via anti-cheat, add to community/totalKm, leave the torch waiting at the drop coords, append torch/history); optional scheduler auto-drop after long idle; admin resetTorch. No-recursion guard on any torch write.
geofence/triggers.ts — onDocumentCreated/onValueWritten for admin-placed NFRs/alerts; find users whose live position is within the radius, look up their users/{uid}.pushToken, and fan out FCM pushes via admin.messaging().sendEachForMulticast(...) (received by expo-notifications). Batched + deduped; respects per-run maxInstances.
missions/takeHome.ts — onCall completeTrail: cross-reference visited stations → matched take-home action (e.g., Reut station → elderly-volunteering link).
antiCheat/validate.ts — speed/teleport sanity checks on position writes; flag impossible km.
auth/onCreate.ts — auth trigger (identity.beforeUserCreated / onInit): on every new user (Google or anonymous) create a users/{uid} profile with default role:"user", capture isAnonymous, name/photo (if Google). Custom claim defaults to user.
lib/secrets.ts — defineSecret for IoT shared secret, FCM keys.
Rules: Firestore + RTDB rules so users write only their own live node; admins (custom claim role:admin) can write NFRs/alerts; IoT path writable only by the function (admin SDK).

Part C — Admin capabilities (secure) — in-app and web dashboard
Admin features exist in two clients sharing one secure backend: the mobile app's (admin) route group (admins in the field) and a web God-Mode dashboard (web/ repo, Vite + React 19 + TS + Firebase JS SDK) deployed to Firebase Hosting. Both use the same secure write-path; neither writes privileged data directly. Admin privilege is enforced at three layers so it is genuinely secure:

Identity: admin = Firebase custom claim role:"admin" in the signed ID token. Granted only server-side via a guarded setAdmin callable (caller must already be admin) + a one-off bootstrap script for the first admin. Clients cannot self-promote; the (admin) UI is gated on the claim, but the UI gate is convenience only — real enforcement is below.
Write path: admins never write privileged data directly. Placing NFRs, firing alerts, editing stations, moving the torch, accruing km all go through onCall Cloud Functions that re-check request.auth.token.role === 'admin' server-side (403 otherwise), enforceAppCheck: true. Functions use the Admin SDK to write.
Rules backstop: Firestore/RTDB rules independently require request.auth.token.role == 'admin' for nfrs/alerts/stations/torch; community/leaderboard are function-only. So even a forged client request is rejected by rules.
In-app admin (mobile) uses react-native-maps (same as user map) subscribing to RTDB live/* — phones and IoT sensor pins render identically. Admin actions call the callables above.

Web Admin dashboard (web/ repo → Firebase Hosting):

Vite + React 19 + TypeScript + Firebase JS SDK ^11 (modular); Google sign-in; the same custom-claim guard (route guard reads getIdTokenResult().claims.role).
Full-screen God-Mode map (@vis.gl/react-google-maps) subscribing to RTDB live/* (onValue) — moving pins for phones + IoT sensors, value-colored station markers.
Side panel: live גלריה photo feed + Leaderboard (runner speeds + community km), Firestore onSnapshot with bounded queries.
Controls: place/edit NFRs, fire GPS/audio alerts to a radius, define take-home, manage stations — all via the same httpsCallable functions (never direct privileged writes).

מירוץ הלפיד — Virtual Torch Relay (exact flow, per the org's spec)
The torch is a shared community km-counter that only ONE person can hold at a time. It is not auto-passed — it waits on the map at the exact GPS point where the last holder dropped it, until any stranger physically reaches that spot and picks it up.

State (RTDB): torch/active: { status:'waiting'|'held', lat, lng, holderId?, holderName?, holderPhoto?, heldSince?, segmentStartLat?, segmentStartLng?, droppedAt }; community/totalKm (the עמותה's shared "km bank"); torch/history/{ts}: { holderId, km, from, to, at }.

Step-by-step (exactly as specified):

הלפיד ממתין על השביל: on app open, the user sees a torch icon on the map at a fixed point — where yesterday's last hiker dropped it (status:'waiting').
איסוף הלפיד: if the user is physically within ~tens of metres of the torch's GPS (on-device distance check vs torch/active.lat/lng), a prominent "קח את הלפיד" button appears. Tapping calls takeTorch callable → an RTDB transaction flips status:'held', sets holderId/heldSince/segmentStart. Locked to her — nobody else in the country can take it; everyone else's map shows "🔥 הלפיד בתנועה עם מטייל אחר".
ההליכה המשותפת: as she walks, the app measures her distance (on-device, throttled) — combining GPS displacement with expo-sensors Pedometer step count for a robust effort signal even when GPS is noisy; a live counter shows "צעדת עם הלפיד 3.2 ק"מ — הוספת למאמץ הקהילתי!". Her live position also moves the torch marker on everyone's map.
סיום מקטע והנחת הלפיד: she taps "הנח את הלפיד" → dropTorch callable: (a) her segment km are added to community/totalKm (validated server-side, anti-cheat — no teleport/impossible speed), shown on the web dashboard "בנק הקילומטרים"; (b) the torch icon "falls" at her exact current coordinates (status:'waiting', new lat/lng) and waits there. Append to torch/history.
הזר שממשיך את המסורת: hours/days later a complete stranger arrives at that area, opens the app, sees the torch waiting at the same point, taps "קח את הלפיד", and accrues km from that point onward. The chain continues.
What the app requires (the org's checklist — all covered):

Torch icon on the map that relocates to wherever the last holder left it.
A stateful button: "קח את הלפיד" ⇄ "הנח את הלפיד".
A simple proximity check (am I close enough to pick it up?).
A distance counter while holding, that sends the total to the server on drop.
Notes: IoT shirt runners can also hold/carry it (ingest sets the holder). Admins can reset/relocate a stuck torch from God-Mode (claim-guarded callable) — e.g. if a holder never drops it; an optional scheduled function auto-drops it at the last known point after a long idle so it never gets stuck on one person. UI: Lottie flame on the marker, haptic on take/drop, a feed post auto-created on each drop so the קהילה follows the journey; the no-trigger-loop guards from the cost section apply to all torch writes.

Data model (summary)
RTDB (live, europe-west1): live/{id}: {lat,lng,speed,ts,source:"phone"|"sensor",name}, torch/active: {holderId, segmentKm, since}, torch/history/*, community/totalKm.

Firestore users/{uid} (secured collection for all users — Google & anonymous):

users/{uid}: {
  uid,
  role: 'user' | 'admin',          // SERVER-controlled via custom claim; client can never set it
  provider: 'google' | 'anonymous',
  isAnonymous: boolean,
  displayName?, photoURL?, email?,  // from Google; null for anonymous
  stationsVisited: string[],        // station ids → drives take-home matching
  totalKm: number,                  // personal accrual (function-written for torch km)
  createdAt, lastActiveAt
}
Doc id = the Firebase Auth uid (so it works identically for Google and anonymous accounts). Rules (below): a user may read all profiles but write only their own doc and only non-privileged fields — role, totalKm, and leaderboard are never client-writable. Created by the auth/onCreate function so every account (incl. anonymous) gets a correct profile.

Other Firestore: feed/{id}, nfrs/{id}: {lat,lng,radius,title,task,active}, alerts/{id}: {lat,lng,radius,message,audioUrl}, stations/{id}, leaderboard (derived), takeHomeRules/{station}.

Security model (hardened — requested)
Sources: App Check for Cloud Functions, Custom claims & rules, Callable functions.

Principle: the client is never trusted. RTDB/Firestore rules are written deny-by-default; all privileged writes (NFRs, alerts, torch transfer, km accrual, IoT pins) go only through Cloud Functions using the Admin SDK, never direct client writes. Each rules file ships as a documented .md in the repo plus the live .rules/.json.

Admin role = Firebase custom claim (not a Firestore field)
Admin is granted by a custom claim role:"admin" set server-side (one-off setCustomUserClaims script / callable guarded by an existing admin). Clients cannot self-promote — the claim lives in the signed ID token.
Rules check request.auth.token.role == 'admin' (Firestore) and auth.token.role === 'admin' (RTDB). Functions check request.auth.token.role server-side.
Cloud Functions — secure header / request handling
Callable functions (onCall): set enforceAppCheck: true and consumeAppCheckToken where one-time; reject !request.auth (401) and request.auth.token.role !== 'admin' for admin ops (403). Auth + App Check arrive in the request automatically — no manual header parsing for callables.
IoT HTTP ingest (onRequest, sensors can't do Firebase Auth): require a secret via header Authorization: Bearer <IOT_SECRET> (or HMAC X-Signature of body+timestamp), IOT_SECRET stored via defineSecret (never in code/env files). Verify with constant-time compare, reject stale timestamps (replay protection), enforce per-device rate limit, restrict CORS, and validate payload shape before any RTDB write.
Enable App Check (Play Integrity on Android, App Attest/DeviceCheck on iOS) so only genuine app instances reach callables and Firestore/RTDB/Storage.
Anti-cheat (antiCheat/validate.ts) runs on the server: reject teleport/impossible-speed position writes so nobody fakes km (the אביב requirement).
Firestore rules (shape — full version in firebaseFunction/firestore.rules + SECURITY.md)
match /users/{uid} {
  allow read: if request.auth != null;
  allow update: if request.auth.uid == uid
        && !request.resource.data.diff(resource.data)
              .affectedKeys().hasAny(['role','totalKm','uid']); // can't self-promote or fake km
  allow create: if request.auth.uid == uid
        && request.resource.data.role == 'user';                // server trigger normally creates it
  allow write: if isAdmin();
}
match /feed/{id}      { allow read: if request.auth != null;
                         allow create: if request.auth.uid == request.resource.data.authorId;
                         allow update, delete: if isAdmin(); }
match /nfrs/{id}      { allow read: if request.auth != null; allow write: if isAdmin(); }
match /alerts/{id}    { allow read: if request.auth != null; allow write: if isAdmin(); }
match /stations/{id}  { allow read: if request.auth != null; allow write: if isAdmin(); }
match /takeHomeRules/{id}{ allow read: if request.auth != null; allow write: if isAdmin(); }
match /leaderboard/{id}{ allow read: if request.auth != null; allow write: if false; } // functions only
function isAdmin(){ return request.auth.token.role == 'admin'; }
// default: allow read,write: if false;
RTDB rules (shape — full version in firebaseFunction/database.rules.json + SECURITY.md)
{ "rules": {
  ".read": false, ".write": false,
  "live": {
    ".read": "auth != null",
    "$id": { ".write": "auth != null && auth.uid === $id && !data.child('source').exists()",
             ".validate": "newData.hasChildren(['lat','lng','ts'])" } },   // phone writes own node only
  "torch":     { ".read": "auth != null", ".write": false }, // take/drop only via callable (Admin SDK)
  "community": { ".read": "auth != null", ".write": false } } }   // km bank accrued by function only
(IoT sensor pins under live/{deviceId} are written by the ingest function via Admin SDK, which bypasses rules — clients can never spoof a source:"sensor" pin.)

Storage rules
Photos under feed/{uid}/… writable only by that uid, readable by signed-in users, with content-type (image/*) + size limits.

Cost efficiency & loop safety (no accidental runaway calls — requested)
Firebase's classic money-leaks are trigger loops (a function writes to the path it triggers on) and unthrottled client traffic (location writes, listener storms). Guards built in:

No function recursion / write-loops:

Functions never write to the path that triggers them. Torch take/drop are callables that write via the Admin SDK; any onValueWritten('torch/active') listener writes only to a different node (e.g. community/totalKm, torch/history) with an idempotency/short-circuit guard (if (before?.status === after?.status && sameCoords) return;) so it can't re-fire itself.
Position-validation triggers write a flag/derived doc, not the watched live/* node.
Where a function must update its own collection, it diffs and early-returns when unchanged.
Throttled client → server traffic:

Location: expo-location uses distanceInterval + timeInterval (e.g. ≥25 m / ≥10 s) so RTDB live/{uid} updates are paced, not per-GPS-tick; debounced in JS too.
Geofence checks run on-device (compare position to cached NFR/alert radii) — a function is only called when a boundary is actually crossed, not continuously.
Torch / km accrual are explicit user actions (single callable), never polled.
Leaderboard/feed use bounded queries (limit, pagination via FlashList), not full-collection listeners; detach listeners on unmount/blur.
Server-side ceilings (defense in depth):

All callables/HTTP funcs set maxInstances + small concurrency/timeoutSeconds/memory so a bug or attack can't fan out to unbounded billable instances.
IoT ingest has per-device rate limiting + payload validation before any write.
A Firebase budget alert (e.g. Cloud Billing budget) is documented in SECURITY.md as a hard backstop, plus Blaze spend caps where available.
App Check blocks non-app traffic from ever reaching billable endpoints.
Verification (added below): an emulator test asserts the torch trigger does not re-fire on its own write, and that a burst of GPS updates collapses to throttled RTDB writes.
